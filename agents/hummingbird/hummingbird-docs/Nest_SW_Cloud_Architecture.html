1. Architecture Overview & Design Philosophy

This document defines the software and cloud architecture for the Hummingbird Nest platform at a level of detail sufficient for engineering implementation. It extends the three-manager architecture described in the Hummingbird Nest Platform Product & Business Plan v9 (Section 8) and addresses the full stack: how users connect, how missions are created and executed, how multiple Nests coordinate, how video and telemetry flow, and how all of it is secured.

1.1 The Fundamental Question: Cloud vs Edge

The central architectural question is: where does the intelligence live?

There are three possible architectures:

ArchitectureLatencyMulti-NestOfflineComplexity
Cloud-Centric
All coordination in the cloud		50–200ms+EasyNoLow
Edge-Only
Everything runs on each Nest		<10msHardYesHigh
Hybrid Edge-Cloud
Real-time at edge, coordination in cloud		<10ms localYesDegraded modeMedium

1.2 Hybrid Edge-Cloud Architecture (Decision)

Architecture Decision: The Hummingbird Nest platform uses a Hybrid Edge-Cloud architecture. Real-time drone control and autonomy run at the edge (on each Nest's onboard computer). The cloud provides authentication, multi-Nest coordination, mission persistence, data recording, and remote access. The system is designed to operate fully autonomously at the edge if cloud connectivity is lost.

Why not cloud-centric? A round-trip to even the nearest cloud region adds 30–100ms of network latency before any processing. For manual drone control where we need <30ms total latency, the cloud cannot be in the control loop. Additionally, public safety operations frequently occur in areas with degraded cellular connectivity. The system must function without the cloud.

Why not pure edge? Multi-Nest coordination requires a common coordination point. Without a central service, each Nest would need to discover and communicate directly with every other Nest — creating an O(n²) mesh that is fragile and complex. A lightweight cloud coordination layer makes multi-Nest operations vastly simpler and more reliable.

The hybrid approach gives us:

  • Sub-10ms local latency for drone control (edge)
  • Full offline capability for single-Nest missions (edge)
  • Scalable multi-Nest coordination (cloud)
  • Secure remote access from any device (cloud)
  • Persistent storage for mission data, video, and analytics (cloud)

1.3 Guiding Principles

  1. Edge-first: Every real-time decision is made at the edge. The cloud is never in the critical control path.
  2. Graceful degradation: Cloud loss = reduced features (no multi-Nest, no remote access), but single-Nest operations continue uninterrupted.
  3. Drone-source agnostic: The Mission Manager allocates drones from a unified pool. It does not know or care which physical Nest a drone belongs to. Travel time is the only differentiator.
  4. Zero-trust security: Every connection is authenticated and encrypted. No implicit trust between any components.
  5. Latency-aware routing: The system dynamically selects the lowest-latency path for each operation type.
  6. Open architecture (MOSA-aligned): The system is built on open, industry-standard protocols and interfaces aligned with the Department of Defense Modular Open Systems Approach (MOSA). All major interfaces use non-proprietary standards: ROS 2 (middleware), DDS (inter-process communication, OMG standard), MAVLink (drone communication, open protocol), WebRTC (video streaming, W3C standard), MQTT (cloud messaging, OASIS standard), and REST/WebSocket APIs (external integration). This ensures interoperability, avoids vendor lock-in, and enables integration with government-directed C2 architectures.
  7. Operator-on-the-loop / operator-in-the-loop: The system supports both autonomous execution with operator oversight (on-the-loop) and direct manual control of individual drones (in-the-loop). Operators can seamlessly transition any drone between these modes during a mission — taking manual control for a closer look, then releasing the drone back to autonomous swarm operation.

1.4 TAK-Native Interface & C2 Integration

1.4.1 The Nest Interface as a TAK-Class System

The Hummingbird Nest's native mission interface is fundamentally a TAK-class situational awareness and command platform. The interface shares the same core design philosophy as the Team Awareness Kit (TAK) ecosystem — the DoD's premier tool for real-time tactical situational awareness, used by over 500,000 military, law enforcement, and emergency response operators worldwide [Breaking Defense, Nov 2025]. Operators familiar with ATAK (Android), WinTAK (Windows), iTAK (iOS), or WebTAK will find the Nest interface immediately recognizable.

The Nest interface provides all of the foundational capabilities expected of a TAK-class system:

1.4.2 Hummingbird-Specific Capabilities Beyond Standard TAK

While the Nest interface provides a TAK-familiar operational experience, it extends beyond standard TAK with capabilities specific to autonomous swarm management that no existing TAK client or plugin currently offers:

  • Autonomous swarm orchestration: Click-to-command mission planning with automated coverage analysis — the operator draws a boundary and the system automatically identifies optimal drone positions, coverage cones, and rotation schedules. This level of automated multi-asset coordination goes significantly beyond TAK's UAS plugin capabilities, which are designed for single-drone observation
  • Drone rotation visualization: Real-time dashboard showing the full lifecycle of every drone in the swarm — active on mission, returning to base, in capture sequence, charging, queued for relaunch. No TAK client provides this level of fleet lifecycle visibility
  • Multi-tier manual control integration: Seamless transition between autonomous swarm operation (operator-on-the-loop) and direct manual stick control of any individual drone (operator-in-the-loop), with latency-adaptive control tier selection. The operator can take over a drone for a closer look, then release it back to the swarm — all within the same interface
  • Automated launch & capture monitoring: Live camera feeds of the robotic arm capture sequence, launch status, and charging bay status — the physical drone lifecycle is visible alongside the tactical picture
  • Swarm health management: Aggregate fleet health metrics (battery distribution, communication link quality per drone, sensor health, rotation efficiency) displayed as an operational overlay
  • Sensor fusion display: Multi-drone sensor data (thermal, visual, environmental) composited into a single unified geographic overlay, providing simultaneous multi-angle situational awareness that a single-drone TAK UAS feed cannot match
  • 3D scene reconstruction: Post-mission cloud-based photogrammetric 3D reconstruction from multi-drone imagery, enabling operators to explore the operational area in three dimensions after the mission — conceptually aligned with VR-TAK (VTAK) style 3D visualization
Design philosophy: The Nest interface is designed so that any operator trained on TAK can sit down and immediately understand the operational picture. The additional swarm-specific capabilities are layered on top of the familiar TAK-class foundation rather than replacing it. This minimizes the learning curve for military and public safety operators who already use TAK daily.

1.4.3 TAK Server Integration (Cursor on Target Protocol)

The Nest provides full interoperability with the TAK ecosystem through native Cursor on Target (CoT) protocol support. The Nest can connect to any standard TAK Server and exchange data bidirectionally with ATAK, WinTAK, iTAK, WebTAK, and TAKX clients:

Data TypeNest → TAK (Published)TAK → Nest (Consumed)
Asset positionsAll drone positions as CoT events (real-time BFT); Nest container positionFriendly force positions for display on Nest map
Sensor eventsAlerts, detections, and sensor readings as CoT markersExternal sensor data for mission awareness
Video feedsDrone video streams published as TAK video service endpoints—
Mission dataCoverage zones, mission boundaries, waypoints as overlaysMission directives, areas of interest, boundaries from higher command
Chat / messagingNest operator messages to TAK chat channelsMessages from TAK users to Nest operator
Data packagesMission reports, imagery, telemetry logsReference data, maps, overlays from TAK server

Protocol implementation: CoT events are generated using the TAK Protocol Version 1 (Protobuf) format for efficient bandwidth usage over constrained links, with XML CoT fallback for legacy TAK server compatibility. Connection to the TAK Server uses TLS-encrypted channels with certificate-based authentication, consistent with standard TAK security practices.

This integration means that a commander using ATAK on a phone, a command post running WinTAK, and field units on iTAK can all see the Hummingbird drone swarm positions and video feeds on their existing TAK maps — without installing any additional software. The drones appear as standard CoT tracks within the established tactical common operating picture.

1.4.4 General C2 API Layer

Beyond TAK-specific integration, the Nest provides a general-purpose REST and WebSocket API layer for integration with any external C2 system. Third-party systems can connect to the Nest (locally on the Nest WiFi network or remotely via the cloud) to:

  • Subscribe to real-time telemetry, drone positions, and mission status via WebSocket streams
  • Issue mission commands and directives via REST API
  • Access video streams via standard WebRTC or RTSP endpoints
  • Receive sensor alerts and event notifications
  • Query mission history, telemetry archives, and recorded video

Data is available in standard formats (JSON, GeoJSON, CoT/XML). The API is documented via OpenAPI specification, enabling straightforward integration by any development team without proprietary SDKs or specialized tooling.

2. System Layers

The architecture is organized into four layers, each with clearly defined responsibilities and interfaces.

USER INTERFACE

2.1 User Interface Layer (Web Client)

A progressive web application (PWA) that runs in any modern browser on tablets, laptops, or phones. This is the operator's window into the system.

  • Mission Planning: Map-based interface for defining coverage zones, mission types, and parameters
  • Live Dashboard: Real-time telemetry, drone positions, battery states, mission progress
  • Video Viewer: Live drone camera feeds via WebRTC with stream switching
  • Manual Control: Direct flight control with gamepad or on-screen controls (requires local connection)
  • Fleet Overview: Multi-Nest status when multiple platforms are deployed
  • Post-Mission: Review recorded data, video playback, generated reports

Technology: React or similar SPA framework, WebRTC for video, WebSocket for real-time telemetry, Progressive Web App for offline support and installability.

CLOUD SERVICES

2.2 Cloud Services Layer

Lightweight cloud services for coordination, persistence, and remote access. Never in the real-time control loop.

  • Auth Service: User authentication (OAuth 2.0 / OpenID Connect), JWT token issuance, RBAC
  • Mission Coordination Service: Multi-Nest mission creation, drone pool aggregation, mission partitioning
  • Telemetry Ingestion: Time-series storage for telemetry, audit logs, operational history
  • Video Relay & Recording: TURN relay for remote video access, long-term video storage
  • Nest Registry: Tracks all Nests, their status, connectivity, and assigned missions
  • Notification Service: Alerts, mission status updates, system health notifications

Technology: Containerized microservices (Kubernetes), PostgreSQL, TimescaleDB for telemetry, S3-compatible object storage for video, NATS or MQTT for pub/sub messaging.

EDGE / NEST

2.3 Edge Layer (Nest Onboard Computer)

The ruggedized onboard computer in each Nest vehicle. This is where real-time intelligence lives.

  • Mission Manager: Translates operator intent into Swarm Manager directives (per v9 8.1.1)
  • Swarm Manager: Real-time drone assignment, rotation, and coordination (per v9 8.1.2)
  • Ground Manager: Physical operations — arm, launch, capture, charging (per v9 8.1.3)
  • Local Web Server: Serves the operator UI directly over local WiFi (no internet needed)
  • WebRTC Origin: Receives video from drones, serves to connected clients
  • Cloud Sync Agent: Maintains connection to cloud, syncs state bidirectionally
  • WiFi Access Point: Provides dedicated operator WiFi network

Technology: ROS 2 (Humble+) on Ubuntu, DDS for inter-node communication, Nginx for local web serving, GStreamer for video pipeline, custom Cloud Sync Agent.

DRONE

2.4 Drone Layer

Each drone's dual-computer system (Pixhawk + Jetson) as defined in v9 Sections 7 and 8.

  • Pixhawk: Flight control, safety, 900 MHz capture mode
  • Jetson: AI/CV processing, ROS 2 node, LTE/mesh communication, video encoding
  • Communication: LTE (primary), WiFi mesh (backup), 900 MHz (capture), Ethernet (inter-board)

Drones communicate exclusively with their home Nest's onboard computer. They have no direct connection to the cloud or to other Nests.

3. User Access & Authentication

3.1 Authentication Flow

A user on any web-capable device (tablet, laptop, phone) can access the system through two paths:

Path A: On-Site (Local WiFi)

1. Operator connects device to Nest WiFi access point (WPA3-Enterprise)

2. Opens browser → navigates to https://nest-local.hummingbird (mDNS / local hostname)

3. Nest serves the PWA directly from its onboard web server

4. Login screen → credentials validated against locally cached user database

5. JWT token issued by Nest → stored in browser session

6. Full access to mission control, video, and manual flight mode

Key design: The Nest caches an encrypted copy of the user database synced from the cloud. This enables authentication even when the Nest has no internet connectivity. The cache is refreshed whenever cloud connectivity is available.

Path B: Remote (Via Cloud)

1. User opens browser → navigates to https://app.hummingbirdtech.ai

2. Cloud Auth Service authenticates via OAuth 2.0 (username/password or SSO)

3. JWT token issued with user roles and authorized Nest IDs

4. User selects active mission/Nest from dashboard

5. Cloud establishes secure tunnel to target Nest

6. Telemetry streams via WebSocket through cloud relay

7. Video streams via WebRTC (TURN relay if P2P not possible)

8. Manual flight control only available if measured latency < 30ms (typically requires on-site)

The most latency-critical operation is manual drone control. The total loop from operator input to drone response must be under 30ms.

Critical constraint: For manual control, the command path is Operator → Nest WiFi → Nest Computer → Drone. The cloud is never in this path. If the command were routed through the cloud, a minimum of 30–100ms of network latency would be added, making responsive manual control impossible.

For remote access, the cloud acts as a secure relay. The Nest maintains a persistent outbound WebSocket connection to the cloud (avoiding NAT/firewall issues). Remote users connect to the cloud, which multiplexes their requests through the appropriate Nest tunnel.

4.4 Manual Control Tiers & Latency Tolerance

Not all manual control is equal. Different control modes have vastly different latency tolerances. The system offers four tiers of manual control, and the UI automatically enables or restricts tiers based on measured round-trip latency:

TierControl ModeWhat Operator ControlsMax LatencyTypical Use
Tier 1
Direct Stick		Rate / Acro modeAngular rates and throttle directly. No stabilization assist.<30msPrecision maneuvering in tight spaces, aggressive repositioning
Tier 2
Stabilized Manual		Attitude / Angle modeDesired pitch/roll/yaw angles. Flight controller stabilizes. Release sticks = level hover.<80msGeneral manual flying, inspection, tracking
Tier 3
Velocity Command		Position-hold / Velocity modeDesired velocity vector (direction + speed). Drone holds position when sticks centered. GPS-assisted.<150msBVLOS manual, slow repositioning, safe exploration
Tier 4
Point & Go		Waypoint nudgeTap on map to set target. Drone flies autonomously to new position. Operator sets altitude and heading.Any (seconds OK)Remote operation, satellite link, degraded comms
Key insight: The system continuously measures round-trip time to each drone. The UI dynamically enables available control tiers and clearly indicates which tiers are available. If latency degrades during Tier 1 control, the system automatically transitions to Tier 2 or 3 with an audible/visual warning — it never allows the operator to maintain a control mode that the link can't support.

Safety rules across all tiers:

  • Single operator lock: Only one operator can have manual control of a drone at a time. Acquiring control requires explicit confirmation.
  • Heartbeat: Control commands include a heartbeat. If the Nest stops receiving heartbeats for 500ms, the drone transitions to autonomous hover, then RTH after 10 seconds.
  • Geofence enforcement: The Pixhawk enforces hard geofence boundaries regardless of operator commands. Manual control cannot fly a drone outside the mission geofence.
  • Altitude floor/ceiling: Hard limits enforced at the flight controller level.
  • Automatic tier downgrade: If latency spikes above the current tier's threshold for >500ms, the system automatically transitions down one tier and alerts the operator.
  • Input protocol: Tiers 1–3 use WebSocket binary frames at 50Hz. Tier 4 uses standard REST/WebSocket commands.

5. Beyond Line of Sight (BVLOS) Operations

Autonomous swarm missions are inherently BVLOS — drones will routinely operate kilometers from the Nest. The autonomy stack handles this without issue: the Swarm Manager coordinates via LTE, and drones fly their waypoints independently. The challenge arises when an operator wants to manually control a drone at BVLOS range.

5.1 The BVLOS Latency Problem

At BVLOS range, the Nest's WiFi (100–300m range) and 900 MHz link (1–2 km, capture only) are unavailable. The only communication path is LTE cellular. Here's the reality of LTE latency:

NetworkTypical RTTBest Case RTTWorst Case RTT
4G LTE30–60ms20ms100ms+
5G Sub-6 GHz10–30ms8ms50ms
5G mmWave5–10ms~5ms15ms
5G URLLC1–4ms~1ms (theoretical; sub-ms not demonstrated in real-world deployments)5ms

The full latency chain for BVLOS manual control over LTE:

SegmentPathLatency
Operator → NestWiFi to Nest computer (on-site)3ms
Nest → Cell towerNest LTE modem uplink10–30ms
Cell tower → Core network → Cell towerCarrier backbone5–20ms
Cell tower → DroneDrone LTE modem downlink10–30ms
Drone processing + PixhawkJetson → Pixhawk → motors5ms
Total (4G LTE)33–88ms
Total (5G Sub-6)23–58ms
The honest assessment: Over 4G LTE, Tier 1 (direct stick) control at <30ms is not achievable. Tier 2 (stabilized, <80ms) is marginal and unreliable. Tier 3 (velocity command, <150ms) works well. Over 5G, Tier 2 becomes reliable and Tier 1 may be possible in ideal conditions. 100ms is fast enough for Tier 3 velocity-command control, which is the appropriate mode for BVLOS operation. Direct stick control at BVLOS range is unsafe regardless of latency due to lack of situational awareness.

This is not a limitation unique to our system — it is a physics and infrastructure constraint. The solutions below address how to provide the best possible operator experience within these realities, and how to push latency lower when the mission demands it.

5.2 Solution 1: Predictive / Assisted Manual Control (Primary)

This is the most important solution and the one that applies to all BVLOS scenarios regardless of radio technology. The core idea: the drone's onboard Jetson handles real-time flight, while the operator provides intent.

How it works:

Operator: Pushes stick forward (intent: “fly north at 5 m/s”)

↓ 50–100ms over LTE

Jetson (onboard drone): Receives velocity command

• Immediately begins executing: fly north at 5 m/s

• Locally handles obstacle avoidance (LiDAR, stereo vision) at <5ms loop

• Locally maintains altitude, attitude, stability at <5ms loop

• If obstacle detected: autonomously avoids, then resumes operator's velocity vector

• If comms lost: holds position, waits, then RTH

↓

Result: Drone responds to the environment in real-time (<5ms). Operator commands arrive at 50–100ms but the drone is always safe because local intelligence fills the gaps.

Military precedent: This is exactly how MQ-9 Reaper and similar military drones operate over satellite links with 1–2 seconds of latency. The operator provides heading, speed, and altitude commands. The onboard autopilot executes in real-time. The Hummingbird system does this at 10–100x lower latency, with AI-powered obstacle avoidance that military platforms lack.

Predictive display:

The operator's UI shows a predicted drone position based on the last command, compensating for latency. When the operator pushes the stick, the UI immediately shows where the drone will be, not where it was 100ms ago. The Jetson confirms actual position, and the UI reconciles smoothly. This eliminates the “flying through molasses” feeling of high-latency control.

Latency-adaptive behavior:

  • <30ms: Full Tier 1 direct control available. Jetson passes commands directly to Pixhawk.
  • 30–80ms: Tier 2 stabilized. Jetson adds local stabilization layer.
  • 80–150ms: Tier 3 velocity-command with full local obstacle avoidance. Primary BVLOS mode.
  • 150–500ms: Tier 3 with increased local autonomy (wider avoidance margins, slower max speed).
  • 500ms+: Automatic transition to Tier 4 (point-and-go only). Warning to operator.

5.3 Solution 2: Mesh Relay Drones

The Hummingbird system has a unique advantage: it has dozens of drones available. One or more drones can be dedicated as airborne communication relays, extending the Nest's WiFi range by kilometers.

Architecture:

Nest (WiFi AP, 5 GHz)

↓ WiFi 6, ~300m

Relay Drone 1 (hovering at altitude, WiFi mesh node)

↓ WiFi 6, ~300m (air-to-air, better propagation at altitude = 500m+ effective)

Relay Drone 2 (optional, for extended range)

↓ WiFi 6, ~500m

Target Drone (the drone being manually controlled)

HopsRangeAdded LatencyTotal RTT (with Nest WiFi)
1 relay~1 km+3–5ms per hop~12–18ms
2 relays~2 km+6–10ms~18–25ms
3 relays~3 km+9–15ms~25–35ms
Key advantage: With just 2–3 relay drones, we achieve Tier 1 (<30ms) manual control at 2+ km range, entirely bypassing LTE. The relay drones hover at altitude for optimal line-of-sight propagation. The Swarm Manager can autonomously position relay drones to form optimal communication chains.

Relay drone requirements:

  • Standard Hummingbird drone with WiFi mesh firmware mode enabled
  • No special hardware — any drone in the swarm can serve as a relay
  • Dual-band WiFi: one radio for uplink (toward Nest), one for downlink (toward target)
  • Relay drones participate in normal rotation — when one returns for charging, another takes its relay position
  • The Swarm Manager automatically positions relays along the optimal path between Nest and mission area

Tradeoff:

Each relay drone is one fewer drone available for the mission. For a 30-drone swarm, dedicating 2–3 to relay duty reduces mission capacity by ~10%. This is an acceptable cost when low-latency manual control at range is required. The Mission Manager factors relay drones into its capacity planning.

5.4 Solution 3: Private 5G / CBRS

For agencies requiring guaranteed low-latency at range without sacrificing drones for relay duty, the Nest can deploy a private 5G cell using CBRS (Citizens Broadband Radio Service, 3.5 GHz band).

5.5 Solution 4: Multi-Access Edge Computing (MEC)

When using commercial LTE/5G, latency can be reduced by keeping traffic at the cellular network edge rather than routing through the carrier's core network and out to the internet.

  • AWS Wavelength: Compute instances deployed at carrier edge (Verizon, Vodafone). Traffic from the drone's LTE modem reaches compute in <10ms.
  • Azure Edge Zones: Similar concept with AT&T partnership.
  • Concept: A lightweight relay/proxy at the cell edge can forward drone traffic directly to the Nest's LTE connection on the same carrier, avoiding the round-trip to the cloud.
PathWithout MECWith MEC
Drone LTE → Nest LTE30–80ms (via carrier core + internet)10–25ms (stays at edge)
Limitation: MEC only helps when both the drone and the Nest are on the same carrier and in the same cell region. It's also dependent on carrier MEC availability, which is not universal. Treat as an optimization, not a primary strategy.

5.6 Recommended BVLOS Architecture

The Nest and each drone continuously evaluate all available communication links and automatically select the best one. If the primary link degrades, the system falls back instantly:

Link priority (Nest → Drone):

2. WiFi mesh via relay drones → if no relay chain available:

3. Private 5G / CBRS (if equipped) → if out of range:

4. Commercial 5G → if unavailable:

5. Commercial 4G LTE → if unavailable:

6. 900 MHz (limited bandwidth, capture commands only) → if all links lost:

7. Autonomous mode: Drone holds position for configurable timeout, then RTH

The UI displays the active link type and current measured latency for each drone. The operator always knows what connectivity is available and what control tiers are enabled. If a manual control session is in progress and the link degrades, the system transitions control tiers smoothly with clear warnings.

Firmware Modes

Each drone’s WiFi 6 radio supports two mesh firmware modes, selectable per-mission by the Swarm Manager:

  • Relay-only mode (default for BVLOS): Drone participates in mesh routing between Nest and mission drones. No local AP broadcast. Optimized for low-latency control traffic.
  • Relay + AP mode (internet delivery): Drone participates in mesh routing and broadcasts a local WiFi access point on a separate channel or SSID. Ground users connect to this AP; traffic is routed upstream through the mesh chain to the internet source. QoS rules prioritize emergency service traffic and deprioritize bulk downloads.

Internet Source Integration

The internet delivery mission requires an upstream internet connection to relay. Three integration paths are supported:

  • Starlink on Nest vehicle: A Starlink terminal mounted on the Nest vehicle roof or deployable mast connects to the Nest’s onboard computer, which bridges the satellite internet connection into the WiFi mesh network. Provides 50–400 Mbps downlink depending on Starlink plan tier: Lite (50–100 Mbps), Residential (100–200 Mbps), or Max (200–400 Mbps) [CableTV.com, 2026].
  • Starlink on a drone: A Starlink Mini or equivalent compact terminal mounted as a drone payload. The drone hovers at altitude for optimal sky visibility and serves as the origin node of the relay chain. This approach is advantageous when the Nest is positioned in an obstructed location (urban canyon, tree cover) where satellite visibility is limited.
  • Hardline / LTE from Nest: The Nest vehicle connects to a functioning internet source (e.g., fiber drop at an emergency operations center, or its own LTE modem) and bridges that connection into the drone mesh. LTE bandwidth (10–50 Mbps typical) is lower than Starlink but requires no additional hardware. Note: the drone fleet’s own LTE modems are reserved for command-and-control traffic and should not be used as the primary internet source for ground user delivery.

Bandwidth Degradation Model

WiFi mesh relay inherently reduces throughput with each hop due to shared-medium contention and half-duplex forwarding. Approximate throughput model for WiFi 6 in open-air conditions with a Starlink source (~150 Mbps at origin):

Chain LengthApprox. Far-End ThroughputAdded LatencyUsable For
3 drones (~750m)40–70 Mbps+9–15msVideo calls, web, file transfer, streaming
5 drones (~1.25 km)20–40 Mbps+15–25msVideo calls, web browsing, emergency portals
8 drones (~2 km)8–20 Mbps+24–40msVoice, messaging, low-bandwidth web, FEMA portals
12 drones (~3 km)3–10 Mbps+36–60msVoice, messaging, text-based emergency services
Mid-chain bandwidth is higher: These figures represent worst-case throughput at the far end of the chain. Drones closer to the internet source retain proportionally higher bandwidth. A 12-drone chain provides ~40 Mbps at drone 3, ~20 Mbps at drone 6, and ~5 Mbps at drone 12. This gradient naturally serves higher-bandwidth needs near the command post and essential connectivity at the far end.

Swarm Manager Integration

The Swarm Manager handles internet relay chains using the same rotation logic as standard missions, with the following additions:

  • Chain integrity monitoring: Continuously verifies mesh link quality between all adjacent drones in the chain. If a link degrades, repositions drones to restore optimal spacing.
  • Sequential rotation: Replaces relay drones one at a time, never removing adjacent drones simultaneously. The replacement drone establishes mesh links with both neighbors before the outgoing drone is released.
  • Multi-chain allocation: When multiple relay chains are active, the Swarm Manager partitions the drone pool and manages rotation schedules for each chain independently, respecting fleet-wide charging capacity constraints.
  • Concurrent mission support: Relay chain drones are flagged as relay_duty in the drone pool. Remaining drones remain available for surveillance, search-and-rescue, or other concurrent missions.
Fleet capacity consideration: A 10-drone relay chain with continuous rotation requires approximately 15–18 drones allocated to the chain (10 airborne + 5–8 in rotation/charging). A 30-drone fleet can sustain one 10-drone chain while retaining 12–15 drones for concurrent missions. Longer chains or multiple simultaneous chains may require a larger fleet or multi-Nest coordination (Section 6).

5.9 GPS-Denied & Communication-Denied Operations

The BVLOS architecture described above assumes GPS availability for navigation and at least intermittent communication for mission coordination. However, the Hummingbird Nest platform is designed to operate in environments where one or both of these may be degraded or completely denied — a requirement for defense, counter-UAS, urban operations, and disaster response scenarios where GPS jamming, spoofing, or infrastructure destruction may occur.

5.9.1 GPS-Denied Navigation Architecture

Each drone maintains a multi-modal navigation stack running entirely onboard its main computer (Jetson Orin Nano). When GPS is degraded or denied, the system transitions seamlessly to alternative navigation sources without any dependency on ground station or cloud processing. The full sensor complement includes:

  • Inertial Navigation System (INS): The Pixhawk's triple-redundant IMU provides continuous position estimation through accelerometer and gyroscope integration. Immune to external interference but subject to drift accumulation over time.
  • Visual-Inertial Odometry (VIO): Camera-based motion estimation fused with IMU data. Provides drift-bounded positioning in feature-rich environments. Processed on the Jetson GPU.
  • Depth SLAM (Simultaneous Localization and Mapping): Uses depth sensors (stereo vision, and LiDAR when available in P2+) to build 3D environment maps and localize within them. Effective in GPS-denied urban and indoor-adjacent environments.

The fusion algorithm dynamically weights each source based on real-time health metrics — signal quality, confidence scores, drift rates, and feature density. This condition-adaptive approach means the system automatically adjusts its navigation strategy based on environmental conditions rather than following a rigid priority hierarchy.

GPS spoofing detection: The GPS module provides integrity monitoring that can detect spoofing indicators (unexpected position jumps, carrier-phase inconsistencies, anomalous signal patterns). When spoofing is detected, GPS is fully excluded from the navigation solution. When GPS is merely degraded (reduced accuracy, intermittent fix), its weighting is reduced proportionally rather than discarded entirely. The response is always proportional to the severity of the detected anomaly.

5.9.2 Cooperative Swarm Localization

When multiple drones operate in a GPS-denied area, the swarm can achieve better positioning accuracy collectively than any individual drone alone through cooperative localization:

  • Shared error covariance: Each drone broadcasts its position estimate and error covariance matrix over the WiFi mesh network. Neighboring drones incorporate these shared estimates into their own fusion algorithm, mutually reducing position uncertainty across the swarm.
  • RF-based inter-drone ranging: WiFi signal strength measurements between mesh modules provide relative distance estimates between drones. These inter-drone distance constraints, combined with shared position estimates, produce a tighter collective position solution.
  • Additional RF frequencies: Other onboard radio modules may contribute supplementary relative ranging measurements for additional redundancy.

This cooperative approach is particularly valuable in large-area operations where some drones may have partial GPS visibility (e.g., operating at the edge of a jammed zone) while others do not. Drones with better positioning data effectively “anchor” the swarm's position estimate, improving accuracy for all members.

5.9.3 Communication-Denied Mission Behavior

When the communication link between a drone (or the entire swarm) and the Nest is lost, drone behavior is determined by the mission profile. Each mission type includes predefined communication-loss parameters set before launch:

Mission TypeComms-Denied BehaviorData HandlingReturn Condition
ReconnaissanceComplete full mission autonomouslyAll video/sensor data stored onboardMission complete or battery threshold
Area Survey / MappingComplete pre-loaded survey patternAll imagery stored onboard for post-processingSurvey complete or battery threshold
Perimeter WatchMaintain assigned position; continue recordingStore-and-forward on reconnectionBattery threshold or configurable timeout
Interactive / Operator-DirectedHold position; attempt achievable sub-objectivesStore-and-forwardConfigurable timeout then RTH

Store-and-forward data model:

1. Drone receives complete mission plan (waypoints, objectives, geofence, sensor configs) before launch

2. If communication is lost mid-flight, the mission plan cannot be updated — the drone executes its pre-loaded instructions

3. All data (video, sensor captures, telemetry) is recorded to onboard storage throughout the mission

4. Upon return to the Nest or restoration of the communication link, data is automatically transferred for processing, viewing, and archival

5. When communication is available, all data streams in real time to the ground station and connected viewers

5.9.4 Swarm Mesh Resilience in Denied Environments

The swarm's WiFi mesh network provides an additional layer of communication resilience. Even when the primary Nest-to-drone link (LTE) is unavailable:

  • Mesh-connected swarm: Drones maintain inter-drone WiFi mesh links. If any drone in the swarm still has a connection to the Nest (via any channel), it relays commands and telemetry for all mesh-connected drones.
  • Information sharing: Drones share navigation data, health status, and mission progress over the mesh, enabling continued coordinated operations even without a Nest link.
  • Graceful degradation to individual operation: If the WiFi mesh itself cannot be maintained (extreme interference or separation), each drone falls back to fully independent autonomous operation on its pre-loaded mission plan and individual INS/VIO/SLAM navigation.
Architectural cross-reference: The multi-modal navigation hardware (INS, VIO sensors, depth cameras) and cooperative localization algorithms are detailed in the Mechanical & Electronics Architecture document, Section 3.4: GPS-Denied & Degraded Navigation. The swarm degradation hierarchy is detailed in Section 4.3.1: Swarm Operations in Denied Environments. This section addresses the software and communication architecture enabling these capabilities.

6. Multi-Nest Operations

This is the critical architectural challenge: when multiple Nest vehicles deploy to the same incident, the operator should see one unified mission with a single pool of available drones — regardless of which physical Nest they belong to.

6.1 Unified Drone Pool Abstraction

Core principle: The Mission Manager works with drones, not Nests. It receives a pool of available drones (each with capabilities, position, battery state, and travel-time-to-station). It assigns drones to mission tasks. It does not know or care which Nest a drone belongs to. The Nest is an implementation detail of how that drone gets launched, recharged, and maintained.

Each Nest reports its available drones to the coordination layer as a Drone Availability Manifest:

The Mission Manager sees a flat list of drones with these attributes and optimizes assignment based on mission needs, drone capabilities, and travel times.

How it works step by step:

  1. Mission created by operator (via any connected client). Mission definition includes coverage area, mission type, sensor requirements, and priority.
  2. Nest discovery: Cloud identifies which Nests are online, in range, and available. Each Nest publishes its Drone Availability Manifest.
  3. Pool aggregation: Cloud's Fleet Mission Manager aggregates all available drones into a unified pool.
  4. Mission partitioning: Fleet Mission Manager divides the mission area into zones and assigns drones to zones based on proximity, capability, and travel time. Drones are preferentially assigned to zones closest to their home Nest to minimize transit time.
  5. Sub-mission dispatch: Each Nest receives its portion of the mission as a sub-mission: “Your drones cover zone B with these waypoints and sensor modes.”
  6. Local execution: Each Nest's local Mission Manager, Swarm Manager, and Ground Manager execute their sub-mission independently. All real-time drone control is local.
  7. Continuous coordination: Nests report status to cloud. Cloud adjusts assignments if drones go down, if a Nest loses connectivity, or if the mission requirements change (dynamic repartitioning).

6.3 Mission Partitioning & Drone Assignment

The Fleet Mission Manager uses the following algorithm for multi-Nest drone assignment:

  1. Zone decomposition: Mission area is divided into a grid or Voronoi tessellation based on required coverage density.
  2. Travel time calculation: For each drone–zone pair, compute estimated travel time from the drone's home Nest to the zone center.
  3. Cost function: Each assignment has a cost = f(travel_time, battery_consumed_in_transit, sensor_match)
  4. Optimization: Hungarian algorithm or similar assignment solver minimizes total mission cost while satisfying coverage constraints.
  5. Rotation planning: Each Nest's Swarm Manager plans its own drone rotation schedule, accounting for the transit time to/from assigned zones.
Important constraint: A drone always returns to its home Nest for recharging. Cross-docking (landing a drone at a different Nest) is not supported in v1. This simplifies Ground Manager design significantly — each Nest only manages its own drones' physical handling.

6.4 Travel Time Modeling

Travel time is the key variable that differentiates drones from different Nests. The Mission Manager must account for:

  • Outbound transit: Time from Nest to assigned zone (reduces effective mission time)
  • Return transit: Time from zone back to home Nest (must be planned before battery is critical)
  • Effective station time: flight_endurance - outbound_transit - return_transit - safety_margin
  • Wind compensation: Adjust transit estimates based on current wind data (headwind increases return time)

For drones far from their home Nest, effective station time may be significantly reduced. The Fleet Mission Manager factors this in — preferring to assign drones to nearby zones and using distant drones only when local capacity is insufficient.

8. Security Architecture

Mandatory requirement: All data in transit across all system layers must be encrypted. No exceptions. This includes drone video, telemetry, control commands, operator sessions, and inter-service communication. Public safety data is sensitive by definition.

8.1 Encryption In Transit

LinkProtocolEncryption
Browser ↔ Nest (local)HTTPS / WSSTLS 1.3 (self-signed cert, pinned in PWA)
Browser ↔ CloudHTTPS / WSSTLS 1.3 (CA-signed cert)
Nest ↔ CloudMQTT / HTTPSmTLS (mutual TLS, client cert per Nest)
Video streamsWebRTCSRTP (AES-128-CM mandatory per RFC 8827; AES-128-GCM negotiable)
Drone ↔ Nest (LTE)ROS 2 DDSDDS Security (AES-256-GCM authentication + encryption)

8.2 Encryption At Rest

  • Nest onboard storage: LUKS full-disk encryption (AES-256-XTS). Unlock via TPM at boot.
  • Cloud databases: AES-256 at-rest encryption (provider-managed keys or customer-managed via KMS)
  • Video archives: Server-side AES-256 encryption in object storage, with per-mission encryption keys
  • User credentials: bcrypt hashed, never stored in plaintext
  • Drone onboard: Encrypted storage on Jetson. Secure boot chain prevents firmware tampering.

8.3 Zero-Trust Network Model

No component implicitly trusts any other component. Every request is authenticated:

  • Users → System: JWT tokens with short expiry (15 min), refresh tokens (24 hr)
  • Nest → Cloud: Mutual TLS with per-Nest client certificates issued during provisioning
  • Drone → Nest: DDS security with per-drone identity certificates
  • Service → Service: mTLS between all cloud microservices
  • Token scope: JWTs contain the specific Nest IDs and roles the user is authorized for. A token for Nest A cannot be used to access Nest B.

8.4 Drone-to-Nest Security

Drones are particularly sensitive because they are physically deployed in the field. Security measures include:

  • Mutual authentication: Each drone has a unique cryptographic identity (X.509 cert stored in secure element). The Nest verifies the drone's identity before accepting commands or telemetry.
  • Anti-spoofing: A rogue device cannot impersonate a legitimate drone. Certificates are provisioned during manufacturing.
  • Command authentication: All commands to the Pixhawk via 900 MHz use MAVLink v2 signing with per-drone keys, preventing command injection.
  • Geofencing enforcement: Geofence limits are stored locally on the Pixhawk. Even if the Jetson or communication is compromised, the Pixhawk enforces hard boundaries independently.
  • Kill switch: Operator can issue an authenticated emergency shutdown command that is processed at the Pixhawk level, bypassing all software layers.

  2. Phase 2 (P2): Cloud services: auth, remote access, telemetry storage, video relay. Single Nest with remote viewers.
  3. Phase 2.5 (P2.5): BVLOS capability: predictive/assisted manual control, mesh relay drone prototype, extended-range communication testing.
  4. Phase 3 (P3): Multi-Nest coordination. Fleet Mission Manager, unified drone pool, mission partitioning.
  5. Phase 4 (P4): Advanced BVLOS: private 5G/CBRS integration, MEC deployment partnerships, full autonomous BVLOS operations.
Phase 1 is the critical path. A single Nest with local operation is the minimum viable product. Cloud and multi-Nest capabilities are force multipliers built on top of a solid edge foundation.
